# 詳細設計（word_practice）

## 1. 目的
- 基本設計を実装可能な粒度に落とし込み、追加判断なしで開発できる状態にする。

## 2. セッション仕様（固定）
- セッション種別: `practice` / `review`
- 1セッションの出題数: 10問固定
- 入力方式: ローマ字入力
- 問題表示: 漢字主表記
- 問題あたり制限時間: 30秒
- ヒント表示:
  - 10秒: 先頭1文字ヒント
  - 20秒: 先頭2文字ヒント
  - 30秒: タイムアウト（不正解）

## 3. 判定ロジック
### 3.1 正規化
- 入力文字列は前後空白を除去して比較する。
- MVPでは大小文字変換・全半角変換・かなカナ変換は実施しない。

### 3.2 正答条件
- `words.lemma_romaji` と入力文字列が完全一致した場合に正解。
- 不一致またはタイムアウトは不正解。

### 3.3 回答記録
- `practice_answers` に以下を保存する。
  - `session_id`, `word_id`, `input_text`, `is_correct`, `hint_stage`, `response_ms`

## 4. 出題選定ロジック
### 4.1 取得優先順
1. ユーザー登録語
2. ローカル初期語彙
3. インターネット取得語
4. ローカルLLM生成語

### 4.2 通常練習（practice）
- 候補からランダム選出し、同一セッション内の重複出題はしない。
- 候補不足時は上位優先順から補充する。

### 4.3 復習（review）
- `word_stats` とSRS情報を使い、`review_due_at` が現在時刻以前の語を優先。
- 不足分は通常練習の選定ルールで補充。

## 5. SRSロジック（固定）
- レベル範囲: `0..5`
- 正解時: `level = min(level + 1, 5)`
- 不正解/タイムアウト時: `level = max(level - 1, 0)`
- 間隔（日）: `[0, 1, 3, 7, 14, 30]`
- `review_due_at = answered_at + interval[level]`

## 6. 集計ロジック
### 6.1 単語統計（word_stats）
- 更新対象:
  - `attempts` +1（毎回答）
  - `wrong_count` +1（不正解時）
  - `timeout_count` +1（タイムアウト時）
  - `hint_reached_count` +1（ヒント表示到達時）

### 6.2 文字統計（char_stats）
- `input_text` と `lemma_romaji` の差分から関連文字を抽出し、試行/誤答を集計。
- MVPでは単純頻度集計とし、編集距離ベースの精密分類は将来拡張。

## 7. ローカルLLM分析・提案
### 7.1 実行タイミング
- セッション終了後に1回実行する（回答中は実行しない）。

### 7.2 入力データ
- `practice_answers`, `word_stats`, `char_stats`, `words`

### 7.3 出力データ
- `analysis_runs`: モデル、状態、要約
- `word_recommendations`: `word_id`, `score`, `reason`, `source`

### 7.4 最終順位
- LLM提案スコアと統計スコア（誤答率・タイムアウト率・直近失敗）を加重合成する。
- MVPの重み: `rule 0.7 / llm 0.3`

## 8. エラーハンドリング
- DB書き込み失敗:
  - セッションを`failed`として記録し、ユーザーへ再試行を案内。
- 外部語彙取得失敗:
  - `vocabulary_fetch_logs` にエラーを保存し、ローカル語彙に自動フォールバック。
- LLM実行失敗:
  - `analysis_runs.status = failed` で保存し、統計ベース提案のみ表示する。

## 9. 画面I/O（主要）
### 9.1 `/practice`, `/review`
- 入力:
  - 開始操作、ローマ字回答文字列
- 出力:
  - 漢字問題文、ヒント、残り時間、正誤フィードバック

### 9.2 `/sessions/:id/result`
- 入力:
  - セッションID
- 出力:
  - 正答率、平均回答時間、連続正解数、推奨単語リスト、提案理由

## 10. 完了条件（工程5ゲート）
- 判定規則、ヒント閾値、タイムアウト条件が固定されている。
- SRS遷移表と更新ルールが固定されている。
- エラーハンドリング方針が明文化されている。
- 実装担当が追加判断なしで着手できる。
