# 課題発見・方針決定（word_practice）

## 1. 目的
- 企画・構想の前に、解くべき課題と進める方針を明確化する。
- 「何を作るか」より先に「なぜ作るか」「何を優先するか」を固定する。

## 2. 現状把握（As-Is）
- 主な困りごと:
  - リアルタイム入力で特定単語に詰まりやすい。
  - 特に仕事で相手の発話を聞きながらメモする場面で発生しやすい。
- 現在の対処:
  - 他者に頼る、周囲のフォローに依存する。
- 想定インパクト:
  - 仕事中の応答遅延、心理的負担、継続的な自己効力感の低下。

## 3. 問題定義（Problem）
- 発音しづらさ・想起困難によりローマ字変換へ移れず、単語入力が停止する。
- 一般的なタイピング練習では、実利用場面の詰まり改善に直結しにくい。

## 4. 方針（Principles）
- 発話を必須にしない。
- 短時間・高頻度（1回3分、週5回）で継続できる設計を優先する。
- 改善評価は「詰まる回数の減少」を主軸にする。
- MVPは単一ユーザー最適化で素早く検証する。

## 5. 制約・前提
- 初期リリースでは音声機能は実装しない。
- 医療診断/治療を目的としない（練習ソフトとして設計する）。
- 本人ヒアリング結果を最優先の要求として扱う。

## 6. Go / No-Go 判定基準
- Go条件:
  - 課題と目的が一文で説明できる。
  - 初期スコープと非スコープが明確。
  - 継続運用（3分×週5）を妨げる致命的障害がない。
- No-Go条件:
  - 解決対象が曖昧で、改善判定の軸が定まらない。
  - 継続利用を阻害する要因が大きく、短期で対策できない。

## 7. 成果物
- 課題定義メモ
- 方針メモ（優先順位・非優先事項）
- Go / No-Go 判定メモ

## 8. 完了条件（工程0ゲート）
- 課題・方針・判定基準が文書化されている。
- ユーザー（本人）ヒアリングで内容が承認されている。
- 次工程（企画・構想）で扱う前提が固定されている。
