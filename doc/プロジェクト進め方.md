# プロジェクト進め方（word_practice / ウォーターフォール）

## 1. 目的
- 「特定の単語が打てない」課題を改善する練習ソフトを、段階的ではなく工程順に開発する。
- 各工程で成果物を確定し、承認後に次工程へ進む。

## 2. 開発モデル
- 一般的なウォーターフォールを採用する。
- 基本順序は以下とし、原則として前工程に戻らない。
1. 要件定義
2. 基本設計
3. 詳細設計
4. 実装
5. テスト
6. リリース
7. 保守・運用

## 3. 工程定義
### 3.1 要件定義
- 目的、対象ユーザー、機能要件、非機能要件、受け入れ条件を確定する。
- 本プロジェクトでは `doc/要件定義.md` を正本とする。
- 完了条件:
  - 要件の抜け漏れがない
  - スコープ外が明記されている（現時点では音声機能なし）

### 3.2 基本設計
- 画面一覧、画面遷移、機能構成、データ構成を定義する。
- 成果物例:
  - 画面一覧・画面遷移図
  - 機能構成図
  - データ項目定義（苦手単語、セッション結果など）
- 完了条件:
  - 主要画面と主要データの責務が明確

### 3.3 詳細設計
- 入出力、バリデーション、エラーハンドリング、判定ロジックを仕様化する。
- 成果物例:
  - 画面項目仕様
  - API/関数I/O仕様
  - 正答判定・復習出題ロジック仕様
- 完了条件:
  - 実装担当が追加判断なしで開発できる

### 3.4 実装
- 詳細設計に従って実装する。
- 実装ルール:
  - 設計変更が必要な場合は設計書を先に更新する
  - 1コミット1目的を守る
  - コミットメッセージは日本語（`doc/コミットルール.md` 準拠）
- 完了条件:
  - 設計で定義した機能が実装済み

### 3.5 テスト
- 単体テスト、結合テスト、受け入れ確認を順に実施する。
- 観点:
  - 苦手単語登録/編集
  - 練習セッション（出題/回答/判定）
  - 結果集計（正答率/回答時間）
- 完了条件:
  - 重大不具合が解消されている
  - 受け入れ条件を満たしている

### 3.6 リリース
- リリース対象バージョンを確定し、公開可能な状態にする。
- 完了条件:
  - リリースノート作成
  - 既知不具合一覧を明記

### 3.7 保守・運用
- 不具合修正、改善要望、次版要件の収集を行う。
- 次版で要件を変更する場合は、再度「要件定義」工程から開始する。

## 4. 工程ゲート（次工程へ進む条件）
- 要件定義完了ゲート: `doc/要件定義.md` が承認済み
- 基本設計完了ゲート: 画面/データ設計が承認済み
- 詳細設計完了ゲート: 実装可能レベルで仕様が承認済み
- 実装完了ゲート: 実装差分レビュー完了
- テスト完了ゲート: 受け入れ条件達成

## 5. ドキュメント管理ルール
- 正式仕様は `doc/` 配下に集約する。
- 要件変更は必ず `doc/要件定義.md` を先に更新する。
- 調査更新は `doc/失語症調査.md` に追記する。
- 本工程ルールの変更は本ドキュメントを更新する。

## 6. 直近の実行順（ウォーターフォール）
1. `doc/要件定義.md` を最終確定する
2. 基本設計ドキュメントを作成する（画面/データ）
3. 詳細設計ドキュメントを作成する（入力/判定/エラー）
4. 実装に着手する
5. テスト仕様に沿って検証する
