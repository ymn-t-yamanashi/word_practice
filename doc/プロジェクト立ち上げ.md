# プロジェクト立ち上げ（word_practice）

## 1. 目的
- 工程2として、体制・期間・進行ルール・成果物管理方針を確定する。
- 以降工程で判断がぶれない運用基盤を作る。

## 2. 体制
- オーナー: ymn
- 実行担当: Codex（実装/文書更新支援）
- 承認者: ymn
- 意思決定ルール:
  - 工程ゲート承認は ymn が行う
  - 仕様変更は対象ドキュメント更新後に承認する

## 3. 期間・進め方
- 開発モデル: ウォーターフォール
- 現在工程: 2（プロジェクト立ち上げ）
- 直近の優先順:
  1. 工程2の運用ルール確定
  2. 工程3（要件定義）の最終固定
  3. 工程4（基本設計）着手

## 4. ドキュメント運用ルール
- 正式仕様は `doc/` 配下に置く。
- 進捗管理の正本は `doc/進行状況.csv` とする。
- 現在工程の判定は `bash scripts/progress_status.sh` で確認する。
- READMEに影響する変更は同コミットで `README.md` を更新する。

## 5. コミット・レビュー運用
- 1コミット1目的
- コミットメッセージは日本語
- 工程移動時は以下を同時更新:
  - 該当工程ドキュメントの承認記録
  - `doc/進行状況.csv`

## 6. リスク管理（立ち上げ時点）
- リスク: 工程の前後関係が崩れる
  - 対策: 進行状況CSVのステータスを工程順で管理
- リスク: ドキュメントと実態の不一致
  - 対策: 工程移動時に必ず根拠ファイルと判定コマンドを確認

## 7. 完了条件（工程2ゲート）

- 進捗更新ルールと判定手順が明確
- 次工程（要件定義）へ進む運用手順が明文化されている
