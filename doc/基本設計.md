# 基本設計（word_practice）

## 1. 目的
- 要件定義で確定した「タイピングソフト主軸」のMVPを、実装可能な画面・機能・データ構成に落とし込む。
- 主要責務と画面遷移を明確にし、詳細設計の入力とする。

## 1.1 技術前提（DB）
- MVPの永続化DBは SQLite3 を採用する。
- 単一ユーザー運用を前提に、まずはローカルファイルDBで構成する。

## 1.2 技術前提（ローカルLLM）
- 日本語強化の既定モデルは `Llama-3.1-Swallow-8B-Instruct` 系とする。
- 代替候補は `Qwen2.5/3 Instruct` 系とし、同一評価項目で比較可能にする。
- 推論実行はローカル推論ランタイム（例: Ollama / llama.cpp）経由で行う。
- 苦手分析機能では、以下をモデル選定の評価軸とする。
  - 日本語の語彙・文脈理解
  - 指示追従性（提案理由の一貫性）
  - 3分セッション運用での応答時間

## 1.3 技術前提（Docker）
- 開発環境は Docker で再現可能にする。
- Docker Compose で `app` サービスを起動できる前提とする。
- SQLite3 のDBファイルは永続ボリュームに保存し、コンテナ再起動後も保持する。
- 実行コマンド・環境変数・ポートはドキュメント化し、ホスト依存設定を最小化する。

## 2. 対象範囲（MVP）
- 初期問題セットを含む出題管理
- 外部語彙取得（インターネット）とローカルLLM語彙生成
- ローカルLLMによる苦手分析と出題優先度計算
- 練習セッション（自由入力、段階ヒント、タイムアウト）
- 学習履歴と苦手傾向表示
- 練習画面内での結果表示（1ページ完結）

## 3. 画面一覧
- `/`（メイン画面）
  - 練習セッション実行（ローマ字入力）
  - セッション結果表示（正答率、平均回答時間、連続正解数）
  - 右カラムに苦手傾向（7件）を縦表示
- `/settings`（MVP最小）
  - 職業選択、職業ベース問題の自動登録、LLM設定

## 4. 画面遷移
1. `/` を開く
2. 問題エリア中央の「スタート」を押す
3. 同一画面で回答、集計、苦手傾向を確認する
4. 必要時のみ `/settings` で職業・LLM設定を変更する

## 5. 機能構成
- 出題管理
  - 問題候補選定（登録語 + 初期問題セット + 外部取得語 + ローカルLLM生成語）
  - 取得優先順制御（登録語 > ローカル初期語彙 > インターネット取得語 > ローカルLLM生成語）
  - 語彙正規化（重複除去、不適切語フィルタ、難易度タグ付け）
  - オフラインフォールバック（ネット不可時にローカル語彙/LLMへ切替）
  - 復習優先度計算（SRS level / due）
- セッション実行
  - 問題提示（漢字主表記）、ローマ字直接入力表示、リアルタイム判定、ヒント制御
  - タイピング挙動の計測（入力速度、ミスタイプ、停止時間）
- 履歴集計
  - 単語単位の詰まり指標集計（主表示）
  - 文字単位の詰まり傾向集計（補助表示）
- 苦手分析・提案
  - セッション終了時にローカルLLMで苦手傾向を推定
  - 分析出力（苦手タイプ、優先出題語、理由）を生成
  - 統計スコア（誤答率・タイムアウト率・直近失敗）と統合して内部優先順位を確定
- 結果表示
  - 正答率、平均回答時間、連続正解数をメイン画面内に表示
  - 苦手傾向を右カラムに7件表示

## 6. データ構成（最低限）
- `words`
  - 主な項目: `id`, `lemma_kanji`, `reading_kana`, `reading_katakana`, `lemma_en`, `lemma_romaji`, `variants`, `note`, `source_type`, `source_ref`, `fetched_at`, `difficulty_tag`, `inserted_at`, `updated_at`
- `practice_sessions`
  - 主な項目: `id`, `mode`, `status`, `started_at`, `ended_at`, `total_questions`, `correct_count`
- `practice_answers`
  - 主な項目: `id`, `session_id`, `word_id`, `question_index`, `input_text`, `is_correct`, `hint_stage`, `response_ms`, `answered_at`
- `word_stats`
  - 主な項目: `word_id`, `attempts`, `wrong_count`, `timeout_count`, `hint_reached_count`, `srs_level`, `review_due_at`, `last_answered_at`, `updated_at`
- `char_stats`
  - 主な項目: `char`, `related_attempts`, `related_wrong_count`, `updated_at`
- `vocabulary_fetch_logs`
  - 主な項目: `id`, `setting_id`, `source_type`, `status`, `fetched_count`, `error_message`, `executed_at`
- `analysis_runs`
  - 主な項目: `id`, `session_id`, `model_name`, `status`, `summary`, `executed_at`
- `word_recommendations`
  - 主な項目: `id`, `session_id`, `analysis_run_id`, `word_id`, `score`, `reason`, `source`(rule/llm/hybrid), `created_at`

※ ER図は `doc/ER図.md` を参照。

## 7. 責務分離
- UI層:
  - 入力受付と表示更新
- アプリケーション層:
  - 出題選定、判定、SRS更新、集計処理
- データ層:
  - セッション/回答/統計の永続化

## 8. 非機能観点（基本設計）
- 3分運用を維持するため、セッション開始操作は最小化する。
- 途中離脱時も履歴が破損しない保存フローにする。
- 将来拡張（音声、複数ユーザー）を妨げないテーブル分離にする。
- LLM分析は毎回答時ではなくセッション終了時に実行し、応答遅延を抑える。
- Docker上での起動時間と再起動性を確認し、セットアップ手順を固定化する。

## 9. 列挙値方針
- `status` 系と `source_type` 系の許容値は `doc/ER図.md` の「列挙値定義」を正本とする。
- 実装では SQLite3 の `CHECK` 制約またはアプリケーション層のバリデーションで許容値を固定する。

- 主要データの責務が定義されている。
